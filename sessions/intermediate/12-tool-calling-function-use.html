Session 12: Tool Calling & Function Use

⏱ 45 min · 💻 Intermediate · 📝 5 Quiz Questions
📖 Learn 🔬 Lab 📝 Quiz

Learn

What Are Tools?

Tools are functions the model can request to be called. The model does not execute them itself — it outputs a structured tool call request, and you execute the function and return the results. This is the foundation of agentic workflows.

The Tool Calling Loop

The agent loop has five steps:

  1. Send your request with tool definitions (and conversation history)
  2. Decide — The model responds with tool_calls or regular text
  3. Execute — You run each tool call (fetch weather, query a DB, call an API)
  4. Report — You send tool results back as role: "tool" messages
  5. Generate — The model produces the final answer using the tool results

Tool Definition Structure

Each tool is defined with these fields:

  • name — Unique identifier the model uses to call the tool
  • description — Critical! The model uses this to decide when to call it. Be detailed.
  • parameters — JSON Schema describing the expected arguments
  • required — Array of mandatory parameter names

Controlling Tool Usage

The tool_choice parameter controls when the model uses tools:

  • "auto" — Model decides when to use tools (default)
  • "none" — No tool usage allowed
  • "required" — Model must use a tool every turn
  • Specific tool name — Force a particular tool (e.g., {"type": "function", "function": {"name": "get_weather"}})

Parallel Tool Calls

Models can request multiple tools in a single response when parallel_tool_calls is enabled (default: true). This is useful for gathering independent data points simultaneously. The finish reason will be "tool_calls" when the model wants to use tools.

Lab

Build a weather tool calling example with a mock function:

python3 -c "
import requests, json, os

url = 'https://openrouter.ai/api/v1/chat/completions'
headers = {'Authorization': f'Bearer {os.environ[\"OPENROUTER_API_KEY\"]}'}

def get_weather(location, unit='celsius'):
    data = {
        'New York': {'temp': 22, 'conditions': 'Sunny'},
        'London': {'temp': 15, 'conditions': 'Cloudy'},
        'Tokyo': {'temp': 28, 'conditions': 'Humid'},
    }
    info = data.get(location, {'temp': 20, 'conditions': 'Unknown'})
    return {'location': location, 'temperature': info['temp'], 'unit': unit, 'conditions': info['conditions']}

tool_def = [{
    'type': 'function',
    'function': {
        'name': 'get_weather',
        'description': 'Get current weather for a location',
        'parameters': {
            'type': 'object',
            'properties': {
                'location': {'type': 'string', 'description': 'City name'},
                'unit': {'type': 'string', 'enum': ['celsius', 'fahrenheit']}
            },
            'required': ['location']
        }
    }
}]

messages = [{'role': 'user', 'content': \"What's the weather in London and New York?\"}]

# Step 1: Initial call
resp = requests.post(url, headers=headers, json={
    'model': 'openai/gpt-4o-mini:free',
    'messages': messages, 'tools': tool_def
})
data = resp.json()
msg = data['choices'][0]['message']
messages.append({'role': msg['role'], 'content': msg.get('content', '')})

if 'tool_calls' in msg:
    for tc in msg['tool_calls']:
        fn_name = tc['function']['name']
        fn_args = json.loads(tc['function']['arguments'])
        result = get_weather(**fn_args)
        messages.append({
            'role': 'tool',
            'tool_call_id': tc['id'],
            'content': json.dumps(result)
        })
    
    # Don't send tools again on the second call
    resp2 = requests.post(url, headers=headers, json={
        'model': 'openai/gpt-4o-mini:free',
        'messages': messages
    })
    print(resp2.json()['choices'][0]['message']['content'])
else:
    print(msg.get('content', 'No response'))
"

Try this: Add more cities to the mock data and ask about them. Experiment with tool_choice: "required" to force the model to always use the tool.

Quiz

Q1. Who executes the tool call — the model or the developer?

Show answer

b — The model suggests the call, but the developer must execute it and send results back.

Q2. What is the most important field in a tool definition to help the model choose correctly?

Show answer

b — The description tells the model what the tool does and when to use it. A vague description leads to wrong tool choices.

Q3. What does tool_choice: "required" do?

Show answer

b — "required" forces the model to use at least one tool — it cannot respond with plain text.

Q4. What finish reason indicates the model wants to call a tool?

Show answer

c — The finish_reason "tool_calls" means the model wants to invoke one or more tools.

Q5. What is parallel tool calling?

Show answer

b — Parallel tool calls let the model request multiple independent functions at once, which you execute and return together.
