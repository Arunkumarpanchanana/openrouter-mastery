Session 29: Production Deployment

⏱ 40 min · 🚀 Expert · 📝 5 Quiz Questions
📖 Learn 🔬 Lab 📝 Quiz

Learn

Retry Strategies

Production systems must handle failure gracefully. Implement exponential backoff with jitter — each retry waits longer than the last, with random jitter to prevent thundering herd problems. Set a max of 3-5 retries before failing. Use the circuit breaker pattern to stop calling a failing provider entirely after repeated errors.

Latency Optimization

Use the :nitro variant for the fastest available throughput. Set preferred_max_latency thresholds to exclude providers that can't meet your timing requirements. Use provider.sort: "latency" to automatically route to the fastest provider for each request.

Uptime Optimization

Provide model fallbacks — an array of backup models that OpenRouter tries if the primary is unavailable. Use provider fallbacks so if one provider for a model is down, another provider serves the request. Set allow_fallbacks: true to enable automatic fallback routing.

Rate Limit Management

Always respect the Retry-After header when you receive a 429 response. Implement client-side rate limiting to avoid hitting limits in the first place. Batch non-urgent requests during off-peak hours to stay within quota.

Provider-Specific Limits

Different providers have different rate limits. Check per_request_limits in the models data to understand constraints before routing. Some providers may offer higher limits at higher price tiers.

The X-OpenRouter-Metadata Header

Opt into detailed routing information per request by sending this header. You'll receive metadata about which provider handled the request, what fallbacks were attempted, and the routing decision chain — essential for debugging production issues.

Monitoring & Scaling

Use Broadcast for observability and track generation IDs for auditing individual requests. OpenRouter auto-scales on its end — you only need to manage your client-side concurrency. Monitor your error rates and latency trends to detect issues early.

Lab

Build a production-ready request function with retries, fallbacks, and timeout handling:

python3 -c "
import requests, os, time, random

url = 'https://openrouter.ai/api/v1/chat/completions'
headers = {'Authorization': f'Bearer {os.environ[\"OPENROUTER_API_KEY\"]}'}

def production_call(messages, models=None, max_retries=3, timeout=30):
    payload = {
        'messages': messages,
        'max_tokens': 500,
    }
    if models:
        payload['models'] = models  # fallback models
    else:
        payload['model'] = 'openai/gpt-4o-mini:free'

    payload['provider'] = {'sort': 'latency'}  # prioritize speed

    for attempt in range(max_retries):
        try:
            resp = requests.post(url, headers=headers, json=payload, timeout=timeout)

            if resp.status_code == 200:
                return resp.json()

            if resp.status_code == 429:
                retry_after = int(resp.headers.get('Retry-After', 2 ** attempt))
                jitter = random.uniform(0, 0.5 * retry_after)
                print(f'Rate limited. Waiting {retry_after + jitter:.1f}s...')
                time.sleep(retry_after + jitter)
                continue

            if resp.status_code in (502, 503):
                wait = (2 ** attempt) + random.uniform(0, 1)
                print(f'Provider error {resp.status_code}. Retry {attempt+1}/{max_retries} in {wait:.1f}s...')
                time.sleep(wait)
                continue

            raise Exception(f'HTTP {resp.status_code}: {resp.text[:200]}')

        except requests.Timeout:
            if attempt < max_retries - 1:
                print(f'Timeout on attempt {attempt+1}. Retrying...')
                continue
            raise

    raise Exception('Max retries exceeded')

result = production_call([{'role': 'user', 'content': 'What is production deployment?'}])
print(result['choices'][0]['message']['content'])
"

Quiz

Q1. What is the recommended retry strategy for production API calls?

Show answer

b — Exponential backoff with jitter prevents thundering herds while giving the provider time to recover, with a reasonable cap on total retries.

Q2. How do you optimize for latency on OpenRouter?

Show answer

a — Sorting by latency and setting max latency thresholds routes to the fastest providers automatically.

Q3. How do you improve uptime with OpenRouter?

Show answer

b — Fallback mechanisms at the model and provider level ensure your application stays up even when individual providers or models are unavailable.

Q4. What should you do when you receive a 429 (rate limited) response?

Show answer

c — The Retry-After header tells you how long to wait before retrying. Respecting it prevents further rate limiting and allows the system to recover.

Q5. What is the X-OpenRouter-Metadata header used for?

Show answer

b — This header requests detailed routing metadata including which provider was selected, fallbacks attempted, and the routing decision chain.
